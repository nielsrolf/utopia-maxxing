# Utopia Maxxing

Evolutionary optimization of utopia essays using LLMs.

## What This Does

Uses an evolutionary algorithm to iteratively improve utopia essays. Each generation:
1. **Selection**: Pairs essays and asks an LLM which is better (with position-bias randomization)
2. **Crossover**: Pairs essays and asks an LLM to combine the best elements into a new essay
3. The next generation is formed from selection winners + crossover offspring, maintaining population size

Essays are judged on three dimensions: **goodness** (genuinely good for humanity), **specificity** (concrete details), and **plausibility** (realistic path from today).

## What we want

We want to evaluate how LLMs differ with respect to what utopias they think would be optimal, in order to learn about the differences and preferences of different LLMs.

## Usage

```bash
# Download initial essays from Supabase
python download_initial_population.py

# Run evolution (lineage.html is auto-generated at the end)
python evolve.py --model gpt-5 --generations 10
python evolve.py --model claude-sonnet-4-20250514 --generations 5 --run-name my-experiment
python evolve.py --model gpt-5 --generations 10 --population-dir runs/prev-run/gen_005

# Options: --concurrency (default 5), --run-name, --population-dir

# Generate summaries for a run (uses DCache, instant if already cached)
python summarize.py runs/<run-name>

# Regenerate lineage visualization (e.g. after adding summaries)
python lineage.py runs/<run-name>

# Serve a run's visualization
python -m http.server 2943 --directory runs/<run-name>
# Then open http://localhost:2943/lineage.html
```

## Lineage Visualization

Each run gets a standalone `lineage.html` with interactive features:

- **Graph**: Nodes are essays (columns = generations), edges show parent relationships
- **Edge filters**: Toggle visibility of selection winners, eliminated losers, and crossover edges
- **Click a node**: Highlights full lineage (ancestors + descendants) via winning/crossover edges
- **Text panel**: Shows essay content below the graph when a node is clicked
- **Summary / Full Text toggle**: Switch between LLM-generated summaries and full essays
- **Influence mode** (on by default): Node opacity reflects how strongly each node is connected to the selected node. Selection = 100% influence pass-through, crossover = 50% per parent. Influence is summed over all paths so ancestor contributions add up correctly.

## Dependencies

- **localrouter**: Unified LLM client (OpenAI, Anthropic, Google) — all LLM calls go through `localrouter.get_response()`
- **openai**: Used by `summarize.py` for summary generation via `oai.responses.create()`
- **cache_on_disk**: `DCache` for caching summaries across runs
- **supabase**: For downloading initial essays
- **trafilatura**: Web content extraction
- **httpx**: Async HTTP client

## Key Implementation Details

- LLM calls use `localrouter` (`ChatMessage`, `MessageRole`, `TextBlock`, `get_response`)
- Selection randomizes A/B order to reduce position bias
- `make_pairings()` ensures every essay participates in at least one selection and one crossover per generation
- Concurrency is controlled via `asyncio.Semaphore`
- Population size stays constant across generations
- Lineage is tracked in `lineage.json` for full genealogy
- Summaries use `DCache` so identical texts are only summarized once even across different runs
- `evolve.py` auto-calls `lineage.generate_lineage_html()` at the end of a run

## Influence Model

The influence of a seed essay on a later-generation essay is computed by forward-propagating from gen 0:
- **Selection winner**: inherits 100% of the winning parent's influence (loser's influence is lost)
- **Crossover offspring**: inherits 50% from each parent
- Influence is **summed over all paths** — if an ancestor reaches a descendant via multiple routes, contributions add up

This means each child's total incoming influence is always exactly 1.0, and across all N final-generation essays the total influence from all seeds sums to exactly N (the population size). No renormalization is needed.

`analyze.py` implements this in Python (see `compute_influence_forward()`). The same logic is implemented in JavaScript in `lineage.html` (see `computeInfluence()`).

## Analysis Workflow

After running experiments, we produce a qualitative + quantitative analysis comparing runs. This section describes the standard analysis procedure so that repeated experiments (varying prompts, checking path dependence, different seed populations, etc.) produce coherent, comparable results.

### Analysis Tools

- **`analyze.py`**: Quantitative analysis script. Computes influence scores, generates plots to `analysis_plots/`. Edit `RUN_CONFIGS` at the top to specify which runs to compare. Produces:
  - `influence_heatmap.png` — seed essay influence on final generation (normalized per model)
  - `influence_per_model.png` — bar charts of seed fitness per model
  - `influence_trajectories.png` — top 5 seeds' influence over 20 generations per model
  - `diversity_entropy.png` — Shannon entropy of influence distribution over generations (convergence speed)
  - `model_agreement.png` — Pearson correlation matrix between models' influence vectors
- **`blog_post.md`**: Written analysis/blog post (update or create a new one per experiment batch)

### Standard Analysis Procedure

When asked to analyze runs, follow this procedure:

#### Step 1: Quantitative analysis
Update `RUN_CONFIGS` in `analyze.py` to list the runs being compared, then run it. This produces the standard plot set. Verify plots visually (read the PNGs).

#### Step 2: Within-generation diversity (use parallel subagents)
Launch one **Explore subagent per run** to read all essays in the final generation (e.g., `gen_020/*.txt`). Each subagent should:
- Read all essays (at least first 800 chars each)
- Identify **shared elements** across most/all essays (governance mechanisms, proper nouns, narrative structures, metaphors)
- Identify elements appearing in only **some** essays
- Rate overall diversity: monoculture (all near-identical) vs. meaningful variation
- List specific recurring proper nouns and system names with counts

This is the most token-intensive step — each subagent reads ~17 full essays. Run all subagents in parallel.

#### Step 3: Theme emergence tracing (use parallel subagents)
After Step 2 identifies each model's signature themes, launch subagents to trace **when each theme first appeared** during evolution. Each subagent should:
- Grep for key phrases across intermediate generations (gen_001, gen_003, gen_005, gen_008, gen_010, gen_015)
- Read relevant snippets to understand context
- Report the earliest generation where each concept appears and how it evolved

This answers: did the theme come from a seed essay, or was it invented by the crossover process? When did it crystallize into its final form?

#### Step 4: Write the analysis
Combine quantitative results (plots) and qualitative findings (subagent reports) into a blog post. The standard structure is:

1. **Experiment description** — what was varied, what was held constant
2. **Per-model convergence** — what each model's final population looks like, including within-generation diversity
3. **Seed essay fitness** — which seeds survived, with the influence heatmap
4. **Cross-model agreement** — correlation matrix, which models agree/disagree
5. **Convergence dynamics** — entropy plot, trajectory plots, when themes emerged
6. **Interpretation** — what does this reveal about model preferences

### Comparing Across Experiment Batches

When running the same models under different conditions (different prompts, different seeds, restarts for path dependence), the analysis should additionally:
- Compare influence vectors between conditions for the same model (do the same seeds dominate?)
- Compare final-generation themes between conditions (do the same ideas emerge?)
- Track which themes are robust (appear regardless of conditions) vs. fragile (sensitive to initial conditions or prompt wording)

### Tips

- Always run within-generation diversity analysis, even if it seems redundant — the degree of monoculture is itself a finding and varies between models
- The subagent-per-run pattern works well because each run's essays are independent and can be analyzed in parallel
- For theme tracing, grep first to narrow down which generations to read, then read specific files — don't read all generations exhaustively
- When writing the blog post, lead with the surprising findings (e.g., "utopia-lol" being the #1 seed for 3/5 models, or the monoculture problem)
